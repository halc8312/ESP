# 対象理解
ESP は日本向けマーケットプレイスの商品収集・管理・価格表公開・CSVエクスポートを行う Flask アプリであり、認証済みユーザーがショップ/商品/価格表を管理し、一部成果物を公開URLとして外部共有できる。主要リスクは「誤登録」「公開範囲逸脱」「価格/在庫の不整合」「スモーク/監視が本番設定で機能しないこと」に集中する。

# テスト戦略
- 既存の route E2E/CLI smoke をベースラインとして先に実行し、回帰の有無を把握する。
- その後、ローカル起動した実アプリへ Chromium で主要導線を通し、正常系→異常系→境界系→権限/状態遷移→表示/非機能の順に深掘りする。
- UI上の成功表示だけでなく、一覧反映、再読込後整合性、直接URLアクセス、公開画面の露出情報、コンソール/失敗リクエストも併せて確認する。
- 実行不能な外部依存は未実施として残し、代替確認手段と残存リスクを記録する。

# 最重要リスク
1. 公開カタログや価格表から内部情報が漏えるリスク
2. 商品作成/編集/価格更新後の一覧・詳細・公開画面間で不整合が出るリスク
3. CSRF、直接URL、ログアウト後戻るなど認証/セッション境界の破綻
4. 削除/アーカイブ/下書き状態の扱いミスによる公開事故
5. スクレイプ/スモーク/ジョブ可視化が本番相当設定では失敗するリスク
6. モバイル表示、長文、特殊文字、複数画像、連打時の UX 劣化

# 優先順位
| 優先度 | 対象 |
|---|---|
| Critical | 認証、商品作成/更新、価格表公開、公開URL情報露出、エクスポート、スモークCLI |
| High | ショップ切替、テンプレート、検索/ページング、直接URL、ログアウト後挙動、レスポンシブ |
| Medium | スクレイプUI、通知表示、日本語/英語表記、コンソール/失敗リクエスト、境界値入力 |
| Low | 未接続外部サービスの詳細リトライ、複数ブラウザ差分、長時間連続稼働 |

# スコープ内
- ローカル Flask 環境で再現できる Web UI/HTTP 導線
- 既存 pytest E2E/CLI smoke による自動確認
- Playwright による Chromium の手動/探索的 E2E
- デスクトップ幅およびスマホ幅の入口確認
- 成果物としてのテスト設計・実行結果・不具合票作成

# スコープ外
- 実本番URLでの確認
- 実メール送信、Webhook、外部決済、SSO
- 実サイト向けライブスクレイピングの完全疎通
- Safari/iOS/Android 実機固有差分
- 長時間負荷/耐久/大規模同時実行

# 前提条件
- リポジトリの依存関係をローカルへインストール済み
- ローカルDBは `/tmp` 配下の専用 SQLite を使用し既存データを汚染しない
- 検証サーバーは `http://127.0.0.1:5001` で起動
- 既存自動テストはテスト専用設定で CSRF を無効化する

# 使用するアカウント / データ / 依存サービス
| 種別 | 内容 |
|---|---|
| アカウント | ブラウザ上で新規登録するテストユーザー |
| 商品データ | 手動登録商品、テンプレート、価格表、公開トークン |
| DB | `/tmp/esp_e2e_browser.db`, `/tmp/esp_cli_smoke.db` |
| 依存サービス | ローカル Flask、SQLite、インメモリキュー |

# 未確定仕様
- 複数ユーザーロールの正式な権限差分
- スクレイプ失敗時のユーザー向けリトライ仕様
- 大量データ時の性能基準値
- 公開カタログの認可期限/トークン失効運用
- エクスポートCSVの外部システム受入制約の最終期待値

# 制約事項
- 利用可能ブラウザは実質 Chromium のみ
- 外部サイトや Redis が必須のケースは環境未整備だと未実施になる
- 本番用のメール/Webhook/外部認証情報は与えられていない
- Playwright 手動検証は時間制約上、高リスク優先で実施する

# リスクベースでの攻め順
1. 認証とセッション境界
2. 商品手動登録と一覧/詳細整合性
3. 価格表作成・公開URL・公開データ露出
4. エクスポート・公開後の再読込整合性
5. 異常入力・二重送信・直接URL
6. レスポンシブ・表示・コンソール/ネットワークエラー
7. CLI スモークや補助オペレーション導線
