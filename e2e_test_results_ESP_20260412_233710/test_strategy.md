## 対象理解
- 初期調査中。READMEより、ESPはFlaskベースの商品管理・スクレイピング・価格表公開システム。

## テスト戦略
- 既存自動テストで基礎回帰を確認後、ローカル起動したWebアプリに対して主要E2E導線を手動/ブラウザ自動で追う。

## 最重要リスク
- 認証/セッション破綻
- ユーザー分離不足による他人データ参照・操作
- 商品/価格表の保存成功表示と実データ不一致
- 価格表公開での機密露出や表示崩れ
- スクレイピング/外部連携失敗時の見かけ上の成功

## 優先順位
| 優先度 | 対象 |
|---|---|
| Critical | 認証、権限制御、商品CRUD、価格表公開、データ整合性 |
| High | ショップ管理、テンプレート、価格設定、検索/一覧反映 |
| Medium | CSV入出力、アーカイブ/ゴミ箱、公開分析 |
| Low | 外部サイト実連携、長時間安定性、大量データ |

## スコープ内
- ローカル環境で再現可能なUI/API/DB整合性確認

## スコープ外
- 未確定。外部サービス実送信、本番ログ、DB本番参照は原則対象外見込み。

## 前提条件
- 依存パッケージ導入後にローカル起動できること
- テスト用DB/アカウントをローカルで作成できること

## 使用アカウント-データ-依存サービス
- テストアカウント: 作成予定
- テストデータ: 手動投入予定
- 依存サービス: SQLite、ローカルWebサーバー

## 未確定仕様
- ロールの具体的種別
- 外部連携の有効化条件
- 本番同等の制約条件

## 制約事項
- 実環境URL未提示
- 外部連携/メール/Webhook/決済等の実査可否は未確認

## リスクベースでの攻め順
1. ログイン/登録/ログアウト/セッション
2. ショップ作成とユーザー境界
3. 商品作成/編集/再表示/一覧反映
4. 価格表作成/商品追加/公開表示
5. 異常系・境界値・直接URLアクセス
6. 未実施領域の整理

## 実査戦略
| 領域 | 方針 |
|---|---|
| UI | ブラウザ操作で主要導線、入力検証、表示、リロード整合性を確認 |
| API | 認証必須APIや公開エンドポイントの応答整合性を必要箇所で確認 |
| データ整合性 | SQLite参照で一覧・詳細・公開面の反映一致を確認 |
| 権限 | 別ユーザー・直接URL・非公開/他人データ操作を確認 |
| 外部連携 | 実査不能時はBlockedとして代替確認と残存リスクを記録 |
| 非機能入口 | コンソール/失敗リクエスト/操作応答遅延の兆候を確認 |
