## 対象理解
- ESP は、ユーザーごとに商品・ショップ・価格表を管理し、公開カタログを共有する Flask ベースのWebアプリ。
- 商品は `Product` / `Variant` / `ProductSnapshot` に分かれ、一覧・編集・価格表・公開面・analytics に波及する。
- 価格表は公開トークン型で、notes サニタイズ、layout 切替、analytics 記録、為替APIフォールバックを持つ。

## テスト戦略
- 既存 pytest で回帰の土台を確認し、その上でブラウザE2EとSQLite裏取りを組み合わせる。
- リスクが高い順に、認証 → ユーザー境界 → 商品登録/更新 → 価格表公開 → analytics → 異常系 → 外部依存の順で実査する。
- 実査不能領域は止まらず、代替確認・残存リスク・必要追加条件を明記する。

## 最重要リスク
| リスク | 理由 |
|---|---|
| ユーザー分離不足 | 他人の商品/価格表/API更新に到達すると情報漏えい・改ざん事故になる |
| 保存成功表示と実データ不一致 | 商品・価格表は一覧/公開/analytics/CSVへ波及するため影響が広い |
| 公開カタログの機密露出 | `source_url` や `site` の露出は顧客向け公開面として不適切 |
| 外部CDN/API依存 | 編集UIやanalytics表示が外部到達性に依存すると運用環境で劣化しやすい |
| レート制限・セッション破綻 | 認証まわりの抜けは不正試行やログアウト後アクセス事故につながる |

## 優先順位
| 優先度 | 対象 |
|---|---|
| Critical | 登録/ログイン/ログアウト、直接URL、API IDOR、商品登録/更新、価格表公開 |
| High | ショップ管理、価格ルール、テンプレート、notesサニタイズ、analytics |
| Medium | アーカイブ/ゴミ箱、CSV、検索/フィルタ、モバイル、複数タブ |
| Low | 実サイトスクレイピング、多量データ、長時間安定性 |

## スコープ内
- ローカル環境で再現可能な UI / API / DB 整合性確認
- 公開カタログと analytics の反映確認
- ブラウザ console / network 観測
- 既存 pytest 実行結果の確認

## スコープ外
- 本番環境URLに対する実査
- 実メール/Webhook/決済など未提示外部送信
- 実サイトへの本格スクレイピング
- クロスブラウザ/実端末全網羅

## 前提条件
- `pip install -r requirements.txt` 実施済み
- ローカル Flask サーバーが `127.0.0.1:5010` で起動可能
- 専用DBを成果物フォルダ配下に分離して既存ファイルを破壊しない

## 使用アカウント-データ-依存サービス
| 種別 | 内容 |
|---|---|
| ユーザーA | `e2e_user_a` |
| ユーザーB | `e2e_user_b` |
| ショップ | `QA Shop A` |
| 商品 | `テスト商品A 😀` → 更新後 `テスト商品A 更新` |
| 価格ルール | `標準E2E更新` |
| 価格表 | `QA Pricelist 1` |
| 依存サービス | cdnjs (TinyMCE/Sortable), jsDelivr (Chart.js), open.er-api.com |

## 未確定仕様
- 権限ロールが一般ユーザー以外に存在するか
- CSV 入出力の必須フォーマット境界
- スクレイピング失敗時の運用期待値
- 本番でCDN利用が許容前提か、セルフホスト前提か

## 制約事項
- 実環境URL未提示
- ブラウザ実行環境で外部CDN/APIが一部ブロックされる
- そのため、外部依存の完全疎通ではなく「失敗時の崩れ方」中心に評価
- 同一IPでレート制限を踏んだ後は同ブラウザでの再ログインが困難

## リスクベースでの攻め順
1. 既存自動テストでベースライン確認
2. ユーザーAで登録・商品登録・価格表公開までのクリティカルパス実査
3. SQLiteで保存結果・一覧・公開面・analytics を裏取り
4. ユーザーBで直接URL/API を用いた所有権確認
5. ログアウト後戻る、レート制限、重複登録など異常系確認
6. 外部CDN/API不達時の console error / 劣化挙動を確認
7. 未実施項目を coverage gap として残す

## 実査戦略
| 領域 | 方針 | 今回の実施状況 |
|---|---|---|
| UI | 実画面で主要導線、保存、再表示、戻るを確認 | 実施 |
| API | 認証済み/他ユーザーの PATCH を直接叩いて結果確認 | 実施 |
| データ整合性 | SQLiteで user/shop/product/pricelist/page_view を直接照合 | 実施 |
| 権限 | ユーザーBで A 所有リソースに直接アクセス | 実施 |
| 外部連携 | CDN/API 不達時の劣化とフォールバックを観測 | 実施（一部） |
| 非機能入口 | console/network error、外部依存、レート制限を確認 | 実施 |
| モバイル | 画面幅変更での確認 | 未実施 |
| CSV / scrape | サンプル/安全条件不足のため後続 | 未実施 |
