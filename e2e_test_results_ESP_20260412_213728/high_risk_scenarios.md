## データ破壊・金銭影響のあるフロー

### HRS-01: 誤った selling_price によるカタログ公開（HIGH RISK）

**リスク**: `selling_price` が再計算されないまま（BUG-E2E-001）パブリックカタログに公開されると、旧価格で外国顧客に提示され、実際の仕入れ価格との乖離が生じる。

```
影響シナリオ:
  - 仕入れ価格が 10,000円 → 20,000円 に上昇
  - スクレイピングで last_price が更新
  - selling_price が更新されない（バグ）
  - カタログに旧selling_price（例: 15,000円）が表示
  - 顧客が注文 → 実際コスト 20,000円 × 1.5 = 30,000円 で販売すべきところを 15,000円 で販売
  → 損失発生
```

**緊急対応**: BUG-E2E-001 の修正が最優先

### HRS-02: スクレイピングジョブDB不整合（HIGH RISK）

**リスク**: `scrape_jobs` テーブルが本番で未作成・未マイグレーションの場合、全スクレイピングジョブが失敗し商品価格・在庫の更新が停止する。

```
影響シナリオ:
  - 商品が売り切れても scrape_jobs 追跡失敗
  - 在庫なし商品が on_sale のまま表示
  - 顧客が注文 → 在庫なしで対応不能
```

**緊急対応**: 本番環境の Alembic マイグレーション適用確認

### HRS-03: CSVエクスポート内部情報漏洩

**リスク**: Shopify エクスポートCSVに `source_url`（仕入れ先URL）が含まれると、Shopify に取り込まれた後に仕入れ先が競合他社に露見する。

```
確認ポイント:
  - export.py の CSV 出力フィールドに source_url が含まれないこと
  - 現状: テスト (test_export_shopify_generates_csv) Pass だが、
    フィールド一覧の詳細確認は追加テスト推奨
```

### HRS-04: ゴミ箱商品の物理削除後のデータ復旧不能

**リスク**: 物理削除後のロールバック手段がない。

```
確認ポイント:
  - /trash/<id>/delete が物理削除か確認
  - DB バックアップ体制の確認（Render のDB バックアップ設定）
  - 削除前に確認ダイアログが表示されるか
```

## 権限事故の恐れがあるフロー

現状: 主要ルートは @login_required 確認済み

```

現状: app.py の Session 設定確認必要
      SESSION_COOKIE_HTTPONLY=True, SAMESITE=Lax は確認済み
```

## 外部連携失敗時の確認項目

### HRS-08: スクレイピング外部サービス失敗時

```
確認ポイント:
  - Mercari/Snkrdunk/Yahoo へのアクセス失敗時のエラーハンドリング
  - ジョブが失敗した場合に UI でエラーが通知されるか
  - 失敗したジョブの再試行機能の有無
  - 部分失敗（100件中50件失敗）の場合のDB状態

現状: Not Run（外部サービスへのアクセス不可）
```

### HRS-09: Redis/RQ 接続失敗時のフォールバック

```
確認ポイント:
  - Redis 未接続時に inmemory キューにフォールバックするか
  - フォールバック時にユーザーへの通知があるか
  - ワーカーが停止した場合のジョブキューの状態

現状: /healthz レスポンスに queue_backend が含まれることを確認
      queue_backend: "inmemory" → Redis なしでも動作する
```

### HRS-10: DB 接続失敗時

```
確認ポイント:
  - PostgreSQL 接続失敗時に適切なエラーページが表示されるか（500 error.html）
  - pool_pre_ping=True が設定されているため、接続確認は実施される
  - 接続タイムアウト後のリカバリ

現状: pool_pre_ping=True の設定確認済み（database.py）
```
