## 対象理解

### システム概要

ESP (E-commerce Scraping Platform) は、日本の中古品マーケットプレイス（Mercari・Snkrdunk・Yahoo オークション・楽天フリマ等）から商品情報を自動スクレイピングし、オーナーが価格設定・在庫管理・カタログ公開できる B2B 向け Web アプリケーション。

### アーキテクチャ

```
[Flask Web App]
  ├── routes/auth.py        # 認証（ログイン・登録・アカウント管理）
  ├── routes/main.py        # ダッシュボード・インデックス
  ├── routes/products.py    # 商品CRUD
  ├── routes/pricelist.py   # プライスリスト管理
  ├── routes/catalog.py     # パブリックカタログ（token認証）
  ├── routes/scrape.py      # スクレイピングジョブ管理
  ├── routes/archive.py     # アーカイブ
  ├── routes/trash.py       # ゴミ箱
  ├── routes/settings.py    # ショップ設定
  ├── routes/pricing.py     # 価格ルール
  ├── routes/api.py         # 内部API
  └── routes/export.py      # CSV/エクスポート

[Worker (RQ)]
  └── jobs/scrape_tasks.py  # 非同期スクレイピングジョブ

[DB] PostgreSQL (本番) / SQLite (テスト)
[Cache] Redis (RQ バックエンド)
[External] Mercari / Snkrdunk / Yahoo / 楽天フリマ スクレイピング
```

### 認証モデル

- Flask-Login によるセッションベース認証
- パスワードハッシュ（Werkzeug `generate_password_hash`）
- ログイン試行: 5回/15分 のレート制限（in-memory）
- CSRF: Flask-WTF `CSRFProtect`
- セッションクッキー: HttpOnly=True, SameSite=Lax

### データモデル主要エンティティ

| モデル | 説明 |
|--------|------|
| User | システムユーザー |
| Shop | ユーザーが管理するショップ |
| Product | スクレイピング取得商品 |
| Variant | 商品バリアント（在庫・SKU） |
| ProductSnapshot | スクレイピングスナップショット履歴 |
| PriceList | 公開カタログ（token付き） |
| PriceListItem | プライスリストに含まれる商品 |
| PricingRule | 価格計算ルール（マージン・送料・手数料） |
| ScrapeJob | スクレイピングジョブ追跡 |
| DescriptionTemplate | 商品説明テンプレート |

## テスト戦略

### アプローチ

1. **リスクベーステスト**: セキュリティ・データ整合性・金銭影響のある機能を最優先
2. **ブラックボックス + グレーボックス**: HTTPレスポンス確認 + ソースコードレビュー
3. **境界値分析**: 入力フィールドの最小値・最大値・NULL・特殊文字
4. **状態遷移テスト**: 商品ステータス変化（公開→アーカイブ→削除）
5. **セキュリティテスト**: IDOR・CSRF・XSS・認証バイパス・直接URLアクセス
6. **探索的テスト**: 仕様書外の熟練QAが疑うポイント

### テストレベル

| レベル | ツール | 対象 |
|--------|--------|------|
| Unit | pytest | サービス・ユーティリティ関数 |
| Integration | pytest-flask | Flask ルート・DB操作 |
| E2E (API-level) | pytest + Flask test client | 主要ユーザーフロー |
| E2E (Browser) | Playwright | UI操作・JavaScript依存機能 |
| Security | 手動 + コードレビュー | 権限・入力サニタイズ・情報漏洩 |

## 最重要リスク

| # | リスク | 根拠 | 影響 |
|---|--------|------|------|
| R-01 | selling_price 再計算漏れ | テスト失敗で確認済み | 商品が誤った価格で公開される |
| R-02 | ScrapeJob テーブル未作成 | テスト失敗で確認済み | ジョブが実行失敗・追跡不能 |
| R-03 | パブリックカタログへの内部情報漏洩 | source_url / site フィールド | 仕入れ先が競合他社に露見 |
| R-04 | IDOR（他ユーザーデータへのアクセス） | URLにIDが含まれる | データ漏洩・改ざん |
| R-05 | ログアウト後の戻るボタン操作 | セッション残存 | 他人のデータ参照 |
| R-06 | 二重クリック・多重送信 | POSTへの重複リクエスト | 商品重複・ジョブ重複 |
| R-07 | スクレイピング失敗時の画面崩れ | 外部API依存 | UXの破綻・ユーザー混乱 |
| R-08 | 非同期テストカバレッジ欠如 | pytest-asyncio 未インストール | バグ未検出リスク |

## 優先順位

### Critical（リリースブロッカー）

- 認証・認可（ログイン・セッション・権限制御）
- パブリックカタログのセキュリティ（内部情報非公開）
- 商品価格計算の正確性
- データ保存の整合性

### High（リリース前に修正必須）

- selling_price 再計算バグ
- ScrapeJob スキーマ不整合
- CSRF/XSS 防御
- ログイン試行制限の有効性
- 各ユーザーのデータ分離

### Medium（次スプリントで対応）

- 非同期テストの pytest-asyncio 対応
- DB bootstrap モックのシグネチャ不一致
- 境界値・異常系のカバレッジ強化

### Low（改善要望）

- エラーメッセージの日本語一貫性
- モバイル表示のレイアウト確認
- 大量データ時のパフォーマンス

## スコープ内

- Flask ルート全エンドポイント（認証・商品・プライスリスト・カタログ・スクレイピング・設定・エクスポート）
- セキュリティヘッダー確認
- CSRF 保護確認
- ログインレート制限
- パブリックカタログの情報漏洩チェック
- 価格計算ロジック
- スクレイピングジョブ追跡
- アーカイブ・ゴミ箱フロー
- データ整合性（作成→更新→削除）
- 入力バリデーション（必須チェック・長さ・形式）

## スコープ外

| 除外項目 | 理由 |
|---------|------|
| 実際のスクレイピング実行（Mercari等への接続） | 外部サービスへのアクセス不可・テスト環境制約 |
| Redis/RQ の実動作確認 | Redis サーバーなし |
| Shopify 連携エンドポイント | テスト環境に Shopify API キー不在 |
| メール送信 | SMTPサーバー不在 |
| 本番データベース（PostgreSQL） | テスト用 SQLite のみアクセス可 |
| パフォーマンス負荷テスト | 現環境ではインフラ未整備 |
| iOS/Android ネイティブアプリ | 対象なし（Web アプリのみ） |

---

## 前提条件

1. Python 3.12 実行環境（pytest, Flask, SQLAlchemy インストール済み）
2. SQLite テストデータベース（conftest.py で自動作成・削除）
3. CSRF 保護をテスト用に無効化 (`WTF_CSRF_ENABLED=False`)
4. ソースコードアクセス（グレーボックステスト可能）

## 使用アカウント / データ / 依存サービス

| 種別 | 内容 |
|------|------|
| テストユーザー | conftest.py および各テストで都度作成（UUID付き） |
| テストDB | `tests/.tmp/test_app_<uuid>.db`（各テスト後削除） |
| 依存サービス | なし（外部サービスはモック or スコープ外） |
| スクレイピングデータ | フィクスチャから固定データを使用 |

## 未確定仕様

| ID | 項目 | 未確定内容 | 調査方法 |
|----|------|-----------|---------|
| SPEC-01 | selling_price 再計算タイミング | 原価変更時に自動再計算するか否か（テスト失敗から仕様違いの可能性） | 仕様書・実装者確認 |
| SPEC-02 | 多重ログインの挙動 | 同一ユーザーが複数ブラウザでログインした場合の動作 | 手動確認 |
| SPEC-03 | カタログトークンの有効期限 | is_active=True で永続化。期限切れ機能の有無 | ソースコード確認 |
| SPEC-04 | ゴミ箱の自動削除 | 一定期間後の自動物理削除機能の有無 | ソースコード確認 |
| SPEC-05 | スクレイピングジョブの再試行 | 失敗時の自動リトライ回数・間隔 | scrape_tasks.py 確認 |
| SPEC-06 | パスワードの最大長 | 登録時の最大文字数制限（コード上は未確認） | auth.py 確認 |

---

## 制約事項

1. **外部接続不可**: 実際のスクレイピング・Redis・PostgreSQL にアクセス不可
2. **ブラウザ自動化限定**: playwright は Chromium headless のみインストール済み
3. **非同期テスト**: pytest-asyncio 未インストールのため async 関数テスト不可
4. **本番データなし**: 実データを使った境界値テストは実施不可

## リスクベースでの攻め順

```
1. [Critical] 認証フロー完全性
   → ログイン成功/失敗/レート制限/ログアウト後戻るボタン

2. [Critical] パブリックカタログ情報漏洩チェック
   → source_url / site フィールドが JSON に含まれないか

3. [High] 権限制御・IDOR チェック
   → 他ユーザーのデータへの直接 URL アクセス

4. [High] 価格計算バグ確認
   → selling_price の再計算タイミングと正確性

5. [High] データ整合性（CRUD サイクル）
   → 商品作成→更新→削除→一覧反映

6. [High] セキュリティヘッダー確認
   → X-Content-Type-Options / X-Frame-Options / Referrer-Policy

7. [Medium] 入力バリデーション・異常系
   → 必須未入力・特殊文字・長文・SQL インジェクション兆候

8. [Medium] 状態遷移
   → 商品ステータス変化・アーカイブ・ゴミ箱フロー

9. [Medium] スクレイピングジョブ追跡
   → ジョブ作成・ステータス変化・DB 記録

10. [Low] UX・表示
    → エラーメッセージ・ローディング・モバイル表示
```
