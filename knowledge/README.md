# ナレッジベース — ESP プロジェクト

本ディレクトリは、AIおよび人間が障害・運用ナレッジを体系的に参照・蓄積するためのナレッジリポジトリです。

---

## ディレクトリ構成

```
knowledge/
├── README.md                   ← 本ファイル（体系図・カテゴリ一覧・命名ガイドライン）
├── deploy/                     ← デプロイ・リリース関連の障害・知見
├── production/                 ← 本番環境の運用・障害対応
├── scraping/                   ← スクレイピング処理に関する障害・知見
├── gunicorn/                   ← Gunicorn設定・ワーカー管理
├── worker/                     ← ワーカープロセス管理・並行処理
├── queue-system/               ← キューシステム（scrape_queue等）の障害・知見
├── system/                     ← システム全体・インフラ構成
├── performance/                ← パフォーマンス改善・ボトルネック調査
├── troubleshooting/            ← 横断的なトラブルシューティング事例
└── {カテゴリ}/                  ← 必要に応じて追加
    └── subtopic/               ← サブトピックが必要な場合は追加可
```

---

## カテゴリ一覧

| カテゴリ名         | 対象範囲                                                   |
|--------------------|------------------------------------------------------------|
| `deploy`           | Render/Docker/Gunicornデプロイ、CI/CD、リリース作業         |
| `production`       | 本番環境での運用障害・インシデント対応                       |
| `scraping`         | ECサイトスクレイピングの失敗・セレクタ変更・Bot対策          |
| `gunicorn`         | Gunicornの設定ミス・ワーカー数・タイムアウト設定             |

| `system`           | アーキテクチャ変更・依存関係・環境変数・DB設計               |
| `performance`      | レスポンスタイム改善・メモリ最適化・スループット向上          |

## ファイル命名規則

### フォーマット

```
{YYYYMMDD}_{概要キーワード1}_{概要キーワード2}_{結果}.md
```

### 命名ルール

| 要素               | 説明                                                                 | 例                          |
|--------------------|----------------------------------------------------------------------|-----------------------------|
| `YYYYMMDD`         | 障害発生日（または解消確認日）                                         | `20260307`                  |
| `概要キーワード`   | 障害の主要コンポーネント・ステージ・内容を `_` 区切りで記述（2〜4個）  | `queue_stage0_worker1`      |
| `結果`             | `success`（解消）/ `investigating`（調査中）/ `wip`（対応中）         | `success`                   |

### 命名例

```
20260307_queue_stage0_worker1_success.md     # Stage0キューワーカー障害の解消記録
20260310_mercari_selector_change_success.md  # Mercariセレクタ変更対応
20260315_render_deploy_timeout_wip.md        # Renderデプロイタイムアウト調査中
20260320_db_migration_fail_investigating.md  # DBマイグレーション失敗調査中
```

## ナレッジファイルの記載テンプレート

各ナレッジファイルには以下のセクションを含めることを推奨します。

```markdown
# {タイトル}

## 概要
- 発生日時・環境・影響範囲を簡潔に記述

## 障害発生状況
- 何が起きたか（症状・エラーメッセージ等）

## 調査経緯
- 調査ステップ・仮説・検証内容の時系列記述

## 根本原因
- 特定された根本原因を明記

## 解消対策
- 実施した対策・コード変更・設定変更の内容

## 再発防止・今後の注意点
- 同様の問題を防ぐためのガイドライン・設計注意点

## 関連ファイル・参考情報
- 関連するソースファイル・PR・Issue・外部ドキュメントのリンク
```

---

## ナレッジ追加フロー

1. 障害発生 → 対応中は `{YYYYMMDD}_..._wip.md` または `_investigating.md` で作成
2. 解消確認後 → ファイル名を `_success.md` にリネームし、根本原因・対策を追記
3. カテゴリが既存に合わない場合 → 新規カテゴリディレクトリを作成し、本READMEのカテゴリ一覧を更新
4. 粒度は **1障害につき1ファイル** を基本とする（複数障害をまとめない）

## Stage進行に伴う重要注意事項

### Stage 0（現在）
- `scrape_queue` はプロセス内インメモリシングルトン
- **Gunicorn は必ず `--workers 1 --max-requests 0` で起動すること**
- ワーカー数を増やすとプロセス間でジョブIDが共有されず、ステータス管理が壊れる

### Stage 1以降（将来）
- ワーカー数拡張には **状態永続化設計（DB、Redis等）** が必須
- インメモリ状態に依存したコードは全てリファクタリングが必要
- ジョブIDのユーザー帰属検証もあわせて実装すること（現状未実装）

詳細は [`deploy/20260307_queue_stage0_worker1_success.md`](deploy/20260307_queue_stage0_worker1_success.md) を参照。
